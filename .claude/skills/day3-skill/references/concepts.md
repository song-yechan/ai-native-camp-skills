## 1. Clarify 프로세스 완전 가이드

### 정의

**Clarify** = 모호함을 명확함으로 바꾸는 과정

```
Before: "이메일 정리해줘"
              ↓ Clarify
After:  "Gmail에서 'AI' 키워드가 포함된 뉴스레터를
        찾아서, 핵심 인사이트를 요약하고,
        insights/YYYY-MM-DD.md 파일로 저장해줘"
```

### 왜 필요한가?

**"Garbage In, Garbage Out"**
- 우리 머릿속의 의도는 **모호한 경우가 많음**
- AI는 모호한 입력을 받으면 **추측해서 실행**
- 추측 = 틀릴 가능성 높음
- Clarify로 **추측 대신 질문하게** 만들기

**흔한 모호한 요청 예시**:
- "우리 팀 업무를 개선해줘" → 어떤 팀? 어떤 업무? 개선 기준은?
- "좋은 보고서 만들어줘" → 어떤 주제? 누구를 위한? 어떤 형식?
- "이메일 정리해줘" → 어떤 이메일? 어떻게 정리? 어디에 저장?

### Clarify 3가지 스킬

#### 1. Vague (모호함 해결) — 가장 기본

**언제 사용**: 초기 단계, 요구사항이 모호할 때

**작동 방식**: Claude가 모호한 부분을 찾아서 **질문으로 명확화**

**명령어**: `/clarify-vague`

**예시 흐름**:
```
사용자: "뉴스레터 스킬 만들어줘"
Claude: (Clarify 질문)
  Q1: "어떤 키워드로 뉴스레터를 찾을까요?"
  Q2: "인사이트의 기준이 뭔가요?"
  Q3: "어떤 형식으로 정리할까요?"
  Q4: "출력 파일 위치는?"
```

#### 2. Unknown-Problem (미지의 영역 찾기)

**언제 사용**: 플랜이 구체화됐지만 **뭔가 빠진 것 같을 때**

**작동 방식**: "아는 것 / 모르는 것 / 모르는 것을 모르는 것" 4분면 분석

**명령어**: `/clarify-unknown`

**4분면**:
| | 아는 것 | 모르는 것 |
|---|---------|----------|
| **아는 것** | 확인된 사실 | 질문할 수 있는 것 |
| **모르는 것** | 숨겨진 가정 | **블라인드 스팟** |

→ 블라인드 스팟을 찾아내는 것이 핵심

#### 3. Meta-Level (내용/형식 분리)

**언제 사용**: 형식을 변경하거나 **새 매체를 고려할 때**

**작동 방식**: 만들려는 **내용**(Content)과 담는 **형식**(Form)을 분리해서 검토

**명령어**: `/clarify-meta`

**예시**: "뉴스레터 인사이트 정리"
- 내용(Content): AI 트렌드 인사이트
- 형식(Form): md 파일? Slack 메시지? Notion 페이지? 이메일?
→ 형식을 바꾸면 내용도 달라져야 할 수 있음

### Clarify 사용 타이밍

**플래닝 직후가 최적의 타이밍!**

```
1. 내가 요청을 입력
2. Claude가 플랜(계획)을 만듦  ← Shift+Tab
3. ⭐ 여기서 Clarify! ⭐
4. 플랜 수정
5. 실행
```

**왜 플래닝 직후인가?**
- Claude는 내 입력을 "참"이라고 가정하고 플래닝
- 모호한 입력 그대로 실행하면 모호한 결과
- 플랜이 만들어지자마자 Clarify로 수정하는 게 가장 효율적

### 설치 및 사용

```bash
# 설치
claude plugin add clarify

# 확인
claude plugin list

# 사용
/clarify-vague
# 또는 자연어로
"모호한 부분을 질문해줘"
```

## 2. 좋은 스킬의 설계 원칙

### 스킬이란?

**스킬 = AI에게 주는 업무 레시피**

**비유**: 요리 레시피
- "이 재료들을 이 순서로 조합하면 이 요리가 나와"
- 재료와 순서만 따라하면 **누가 해도 같은 결과**
- 스킬대로 하면 **매번 같은 품질의 결과**

### CLAUDE.md vs Skill

| 구분 | CLAUDE.md | Skill |
|------|-----------|-------|
| 비유 | 업무 매뉴얼 | 요리 레시피 |
| 로딩 | **항상** (매 세션) | **필요할 때만** |
| 내용 | "나는 이런 사람" | "이 일은 이렇게" |
| 예시 | 선호 언어, 작업 스타일 | 주간 리포트 생성 |

**핵심**: 컨텍스트 윈도우는 유한 → 필요한 것만 로딩 (Just-in-time)

### 좋은 스킬의 3가지 조건

#### 조건 1: 목적이 명확하다
- "이 스킬은 **무엇을 해결**하는가?"
- **한 문장**으로 설명할 수 있어야 함

| 나쁜 예 | 좋은 예 |
|---------|---------|
| "이메일 관련 여러 가지를 한다" | "Gmail에서 특정 키워드의 뉴스레터를 찾아 인사이트를 정리한다" |

#### 조건 2: 입력과 출력이 분명하다
- **입력**: 무엇을 받는가? (URL? 파일? 텍스트? 키워드?)
- **출력**: 무엇을 내놓는가? (md 파일? JSON? Slack 메시지?)

```
예시:
  입력: Gmail 검색 키워드 (AI, Tech 등)
  출력: insights/YYYY-MM-DD.md 파일
```

#### 조건 3: 실행 단계가 구체적이다
- "1. 이것을 하고 → 2. 저것을 하고 → 3. 완료"
- **모호함이 없어야** 함
- Claude가 추측하지 않고 따라할 수 있어야 함

| 나쁜 예 | 좋은 예 |
|---------|---------|
| "이메일을 적절히 정리한다" | "1. Gmail에서 키워드로 검색 2. 최근 7일 이메일 중 상위 5개 선택 3. 각 이메일에서 제목, 발신자, 핵심 내용 추출 4. 형식에 맞춰 md 파일 생성" |

## 3. SKILL.md 작성 가이드

### 스킬 폴더 구조

```
.claude/skills/my-skill/
├── SKILL.md          # 필수: 스킬의 본체
├── scripts/          # 선택: 실행할 코드
├── references/       # 선택: 참고 문서
└── assets/           # 선택: 템플릿, 리소스
```

**SKILL.md 하나만 있으면 스킬이 됩니다!** 나머지는 필요할 때 추가.

### SKILL.md 구조

```markdown
---
name: newsletter-curator
description: Gmail에서 뉴스레터를 수집하여 인사이트를 정리하는 스킬
---

# 뉴스레터 큐레이터

## 목적
Gmail에서 뉴스레터를 수집하고 핵심 인사이트를 정리

## 입력
- Gmail 검색 키워드: AI, Tech 등

## 실행 단계
1. Gmail에서 키워드 포함 이메일 검색
2. 각 이메일에서 핵심 내용 추출
3. 인사이트 정리하여 md 파일 생성

## 출력
- 파일: insights/YYYY-MM-DD.md
```

### frontmatter 작성

```yaml
---
name: skill-name          # 영문 소문자 + 하이픈
description: 스킬 설명     # 한 문장으로
---
```

- `name`: 영문 소문자와 하이픈만 사용
- `description`: 한 문장으로 스킬이 무엇을 하는지 + 언제 호출될지 서술

### 로딩 방식

| 방식 | 예시 |
|------|------|
| 직접 호출 | `/newsletter-curator` |
| 자동 매칭 | "뉴스레터 정리해줘" → description 매칭 |
| 명시적 호출 | "newsletter-curator 스킬 실행해줘" |

## 4. 스킬 체이닝 개념

### 정의

**스킬 체이닝** = 하나의 스킬 결과를 다른 스킬의 입력으로 연결

### 비유: 공장 생산라인

```
[원자재] → [1공정] → [2공정] → [3공정] → [완성품]
```

각 공정이 **독립적으로 존재**하면서, 앞 공정의 결과물을 다음 공정이 받아서 처리

### 실전 예시

#### 예시 A: 뉴스레터 → 주간 리포트
```
[Gmail] → [newsletter-curator] → [insight-tagger] → [weekly-digest] → [Slack/Notion]
           뉴스레터 수집         주제별 태그 분류    주간 리포트 통합    팀 공유
```

#### 예시 B: 고객 피드백 → 기획 문서
```
[문의 수집] → [feedback-classifier] → [priority-scorer] → [prd-drafter]
              유형별 분류              우선순위 점수 부여    PRD 초안 생성
```

#### 예시 C: 미팅 → 후속 작업
```
[녹취록] → [meeting-summarizer] → [action-extractor] → [task-creator] → [followup-drafter]
           회의 요약               액션 아이템 추출     태스크 생성       팔로업 이메일
```

### 체이닝 설계 원칙

#### 원칙 1: 한 스킬 = 한 가지 일
- "이것도 하고 저것도 하는" 스킬 ❌
- "이것만 잘하는" 스킬 여러 개 ⭕

#### 원칙 2: 입출력 맞추기
- 스킬 A의 출력 형식 = 스킬 B의 입력 형식
- md 파일 → md 파일, JSON → JSON

#### 원칙 3: 중간 결과물 저장
- 각 단계의 결과를 **파일로 저장**해두면
- 문제 발생 시 **디버깅이 쉬움**

### 실행 방법

**방법 1: 한 번에 연결**
```
"newsletter-curator 실행하고, 그 결과를 weekly-digest에 넘겨줘"
```

**방법 2: 단계별로 (추천)**
```
1. "/newsletter-curator" 실행
2. 결과 확인
3. "이 결과를 weekly-digest로 정리해줘"
```

→ 처음에는 단계별로 하면서 중간 결과 확인하는 게 좋음

## 5. 플랜 모드 사용법

### 플랜 모드란?

Claude가 **바로 실행하지 않고 계획을 먼저 보여주는** 모드

### 전환 방법

| 방법 | 사용법 |
|------|--------|
| 키보드 단축키 | `Shift + Tab` (플랜 모드 ON/OFF 토글) |
| 자연어 | "플랜부터 만들어줘", "일단 계획만 세워줘, 실행은 하지 마" |

### 활용 흐름

```
1. 요청 입력 + Shift+Tab (또는 "플랜부터 만들어줘")
2. Claude가 계획을 보여줌
3. 계획 검토 → Clarify로 모호함 제거
4. 수정된 계획 확인
5. 승인 → 실행
```

### 왜 플랜 모드를 쓰는가?

- 바로 실행하면 **되돌리기 어려움**
- 계획을 먼저 보고 **수정할 기회**
- **Clarify와 함께** 쓰면 가장 효과적

### Day 3의 핵심 워크플로우

```
플랜 모드로 계획 생성
    ↓
/clarify-vague로 모호함 제거
    ↓
플랜 수정
    ↓
실행
```

## 6. 의사결정 로그 파일 가이드

### 정의

AI와 대화하면서 **결정한 내용을 기록하는 문서**

### 왜 필요한가?

- AI와 대화하면서 많은 결정을 내림
- 시간이 지나면 **왜 그런 결정을 했는지 잊어버림**
- 스킬 수정할 때 맥락 파악에 도움
- 팀원에게 인수인계할 때 유용

### 기록할 내용

1. 날짜 및 작업 제목
2. **결정한 사항** (키워드, 기준, 형식 등)
3. **보류한 사항** (나중에 할 것)
4. 논의 중 나온 대안들
5. 최종 선택한 이유

### 예시 형식

```markdown
## 2026-02-21 뉴스레터 스킬 기획

### 결정 사항
- 키워드: AI, Tech, Startup으로 결정
- 인사이트 기준: 새로운 관점 + 실행 가능성
- 출력 형식: 요약 + 인사이트 + 액션 아이템

### 보류
- 주간 통합 리포트는 v2에서
- Notion 연동은 나중에

### 논의 내용
- 일일 vs 주간 → 일일로 시작, 필요하면 주간 추가
```

### 저장 위치

- `decision-logs/` 폴더에 날짜별 저장
- 또는 CLAUDE.md에 간단히 기록

## 7. 스킬 제작 3단계

### Step 1: 문제를 스킬 언어로 번역 (10분)

**자연어 문제** → **입력 → 처리 → 출력**으로 구조화

```
자연어: "뉴스레터가 너무 많아서 다 못 읽어요.
        중요한 인사이트만 골라서 정리하고 싶어요."

↓ 번역

스킬 언어:
  입력: Gmail에서 특정 키워드가 포함된 이메일
  처리: 각 이메일에서 핵심 인사이트 추출
  출력: insights/YYYY-MM-DD.md 파일
```

**팁**: "입력 → 처리 → 출력"으로 정리하면 스킬이 보인다

### Step 2: Clarify로 모호함 제거 (15분)

```
1. 플랜 만들기 (Shift+Tab)
2. /clarify-vague 실행
3. Claude의 질문에 답변
4. 플랜 수정 확인
```

### Step 3: SKILL.md 작성 + 테스트 (25분)

```
1. Claude에게 스킬 생성 요청
   "지금까지 정리한 내용으로 [스킬명] 스킬을 만들어줘."
2. SKILL.md 확인 (frontmatter, 입출력, 실행 단계)
3. 테스트 실행
4. 결과 확인 → 수정 요청 → 반복
```

### 핵심 마인드셋

> "완벽하게 만들려고 하지 마세요. 일단 만들고, 쓰면서 고치면 됩니다."
>
> 스킬은 한 번에 완성되지 않습니다. v1 → v2 → v3... 계속 진화합니다.

## 8. 비개발자 FAQ

### Q: Clarify가 너무 어려워요
"Clarify는 결국 '질문하기'입니다. Claude에게 '모호한 부분을 질문해줘'라고 하면 Claude가 알아서 질문합니다. 여러분은 그 질문에 답하기만 하면 돼요."

### Q: SKILL.md를 직접 작성해야 하나요?
"아니요! Claude에게 '이런 스킬 만들어줘'라고 하면 Claude가 SKILL.md를 작성해줍니다. 여러분은 결과를 확인하고 수정 요청만 하면 됩니다."

### Q: 스킬이 인식이 안 돼요
"세 가지를 확인하세요: 1) 폴더 위치가 `.claude/skills/` 안에 있는지, 2) SKILL.md 파일이 있는지, 3) frontmatter가 제대로 작성됐는지. 그래도 안 되면 터미널을 재시작하세요."

### Q: 결과가 매번 달라요
"SKILL.md의 실행 단계가 모호할 가능성이 높습니다. '적절히', '알아서' 같은 표현이 있으면 Claude가 매번 다르게 해석해요. 단계를 더 구체적으로 적어보세요."

### Q: Shift+Tab이 안 먹어요
"키보드 설정에 따라 다를 수 있어요. 대안으로 Claude에게 '플랜부터 만들어줘' 또는 '일단 계획만 세워줘, 실행은 하지 마'라고 말하면 같은 효과입니다."

### Q: 내 문제가 스킬로 만들기 적합한지 모르겠어요
"Claude에게 물어보세요! '이 문제를 스킬로 만들 수 있을까? [문제 설명]'이라고 하면 Claude가 가능한지, 어떻게 접근하면 좋을지 알려줍니다."
