# Day 3: 해결 — 내 문제 해결 스킬 완성

## 1. Day 3 개요

| 항목 | 내용 |
|------|------|
| 테마 | 해결 (Solve) — "내 문제 해결 스킬 완성" |
| 핵심 메시지 | **좋은 스킬은 좋은 질문에서 나온다. Clarify로 의도를 명확히 하고, 스킬로 자동화하라.** |
| 총 소요 시간 | 약 150분 (2시간 30분) |
| 사전 과제 | Day 2에서 만든 Context Sync 스킬 확인 + Gmail MCP 연결 |
| 산출물 | 문제 해결 스킬 v1 (각자의 문제를 해결하는 스킬) |

### 학습 목표

1. Clarify 프로세스의 중요성을 이해한다 (모호함 제거)
2. 좋은 스킬의 설계 원칙 3가지를 적용할 수 있다
3. 실제 워크플로우를 스킬로 제작할 수 있다
4. 스킬 체이닝 개념을 이해하고 설계할 수 있다
5. Day 1에서 정의한 문제를 해결하는 스킬 v1을 완성한다

#### 슬라이드 2-4: 좋은 스킬의 3가지 조건
```
좋은 스킬의 3가지 조건

┌─────────────────────────────────────────────────────┐
│  1️⃣ 목적이 명확하다                                │
│     "이 스킬은 무엇을 해결하는가?"                  │
│     → 한 문장으로 설명할 수 있어야 함               │
├─────────────────────────────────────────────────────┤
│  2️⃣ 입력과 출력이 분명하다                         │
│     입력: 무엇을 받는가? (URL? 파일? 텍스트?)       │
│     출력: 무엇을 내놓는가? (md 파일? 메시지?)       │
├─────────────────────────────────────────────────────┤
│  3️⃣ 실행 단계가 구체적이다                         │
│     "1. 이것을 하고 → 2. 저것을 하고 → 3. 완료"    │
│     → 모호함이 없어야 함                            │
└─────────────────────────────────────────────────────┘
```

#### 슬라이드 2-5: 문제는 "모호함"
```
문제는 "모호함"

❌ "우리 팀 업무를 개선해줘"

❌ "이메일 정리해줘"

┌─────────────────────────────────────────────────────┐

│                                                     │
│  내 의도가 모호하면 → AI 결과도 모호하다           │
│  내 의도가 명확하면 → AI 결과도 명확하다           │
└─────────────────────────────────────────────────────┘

💡 AI가 못하는 게 아니라, 내가 제대로 안 알려준 것
```

#### 슬라이드 2-8: Clarify의 3가지 스킬
```
Clarify의 3가지 스킬

┌─────────────────────────────────────────────────────┐
│  1️⃣ Vague (모호함 해결)                            │
│     가장 기본. Claude가 모호한 부분을 찾아서 질문   │
│     → 초기 단계에서 가장 많이 사용                  │
├─────────────────────────────────────────────────────┤
│  2️⃣ Unknown-Problem (미지의 영역 찾기)             │
│     "아는 것 / 모르는 것 / 모르는 것을 모르는 것"  │
│     → 놓친 부분이 있을 것 같을 때                   │
├─────────────────────────────────────────────────────┤
│  3️⃣ Meta-Level (내용/형식 분리)                    │
│     만들려는 '내용'과 담는 '형식'을 분리해서 검토   │
│     → 형식을 바꾸거나 새 매체를 고려할 때           │
└─────────────────────────────────────────────────────┘

💡 오늘은 Vague를 중심으로 실습합니다
```

```

│                                                     │
│  1. 내가 요청을 입력                                │

→ Claude는 내 입력을 "참"이라고 가정하고 플래닝함
→ 모호한 입력 그대로 실행하면 모호한 결과가 나옴
→ 플랜이 만들어지자마자 Clarify로 수정하는 게 효율적
```

## 3. 핵심 개념 상세 정리

### 3.1 Clarify 프로세스 완전 가이드

```
[Clarify 프로세스 완전 가이드]

1. 정의
   - Clarify = 모호함을 명확함으로 바꾸는 과정
   - AI에게 명확한 지시를 내리기 위한 핵심 기술
   - "Garbage In, Garbage Out"을 방지

2. 왜 필요한가?
   - 우리 머릿속의 의도는 모호한 경우가 많음
   - AI는 모호한 입력을 받으면 추측해서 실행
   - 추측 = 틀릴 가능성 높음
   - Clarify로 추측 대신 질문하게 만들기

3. Clarify 3가지 스킬
   
   ① Vague (모호함 해결)
      - 가장 기본적인 스킬
      - Claude가 모호한 부분을 찾아서 질문
      - 사용 시점: 초기 단계, 요구사항이 모호할 때
      - 명령어: /clarify-vague
   
   ② Unknown-Problem (미지의 영역 찾기)
      - "아는 것 / 모르는 것 / 모르는 것을 모르는 것"
      - 놓친 부분을 찾아내는 스킬
      - 사용 시점: 플랜이 구체화됐지만 뭔가 빠진 것 같을 때
      - 명령어: /clarify-unknown
   
   ③ Meta-Level (내용/형식 분리)
      - 만들려는 '내용'과 담는 '형식'을 분리
      - 형식을 바꾸면서 모호한 점 발견
      - 사용 시점: 형식을 변경하거나 새 매체를 고려할 때
      - 명령어: /clarify-meta

4. Clarify 사용 타이밍
   1) 내가 요청을 입력
   2) Claude가 플랜(계획)을 만듦 (Shift+Tab으로 플랜 모드)
   3) ⭐ 여기서 Clarify! ⭐
   4) 플랜 수정
   5) 실행

5. Clarify 없이 바로 실행하면?
   - Claude가 내 입력을 "참"이라고 가정
   - 모호한 부분을 추측해서 채움
   - 내 의도와 다른 결과가 나올 확률 높음

6. 설치 및 사용
   설치: claude plugin add clarify
   확인: claude plugin list
   사용: /clarify-vague 또는 "모호한 부분을 질문해줘"
```

### 3.2 의사결정 로그 파일 가이드

```
[의사결정 로그 파일 가이드]

1. 정의
   - AI와 대화하면서 결정한 내용을 기록하는 문서
   - 나중에 "왜 이렇게 했더라?"를 추적할 수 있음

2. 왜 필요한가?
   - AI와 대화하면서 많은 결정을 내림
   - 시간이 지나면 왜 그런 결정을 했는지 잊어버림
   - 스킬 수정할 때 맥락 파악에 도움
   - 팀원에게 인수인계할 때 유용

3. 기록할 내용
   - 날짜 및 작업 제목
   - 결정한 사항 (키워드, 기준, 형식 등)
   - 보류한 사항 (나중에 할 것)
   - 논의 중 나온 대안들
   - 최종 선택한 이유

4. 예시 형식

   ## 2026-02-21 뉴스레터 스킬 기획
   
   ### 결정 사항
   - 키워드: AI, Tech, Startup으로 결정
   - 인사이트 기준: 새로운 관점 + 실행 가능성
   - 출력 형식: 요약 + 인사이트 + 액션 아이템
   
   ### 보류
   - 주간 통합 리포트는 v2에서
   - Notion 연동은 나중에
   
   ### 논의 내용
   - 일일 vs 주간 → 일일로 시작, 필요하면 주간 추가

5. 저장 위치
   - 프로젝트 폴더 내 decision-logs/ 폴더
   - 또는 CLAUDE.md에 간단히 기록
```

### 3.3 좋은 스킬의 설계 원칙

```
[좋은 스킬의 설계 원칙]

1. 목적이 명확하다
   - "이 스킬은 무엇을 해결하는가?"
   - 한 문장으로 설명할 수 있어야 함
   - 예: "Gmail에서 뉴스레터를 수집하여 인사이트를 정리한다"
   
   ❌ 나쁜 예: "이메일 관련 여러 가지를 한다"
   ⭕ 좋은 예: "Gmail에서 특정 키워드의 뉴스레터를 찾아 인사이트를 정리한다"

2. 입력과 출력이 분명하다
   - 입력: 무엇을 받는가?
     - URL? 파일 경로? 텍스트? 키워드?
   - 출력: 무엇을 내놓는가?
     - md 파일? JSON? Slack 메시지? 콘솔 출력?
   
   예시:
   - 입력: Gmail 검색 키워드 (AI, Tech 등)
   - 출력: insights/YYYY-MM-DD.md 파일

3. 실행 단계가 구체적이다
   - "1. 이것을 하고 → 2. 저것을 하고 → 3. 완료"
   - 모호함이 없어야 함
   - Claude가 추측하지 않고 따라할 수 있어야 함
   
   ❌ 나쁜 예: "이메일을 적절히 정리한다"
   ⭕ 좋은 예: 
      "1. Gmail에서 키워드로 검색
       2. 최근 7일 이메일 중 상위 5개 선택
       3. 각 이메일에서 제목, 발신자, 핵심 내용 추출
       4. 형식에 맞춰 md 파일 생성"
```

### 3.4 스킬 폴더 구조 상세

```
[스킬 폴더 구조 상세]

.claude/skills/my-skill/
├── SKILL.md          # 필수: 스킬의 본체
├── scripts/          # 선택: 실행할 코드
├── references/       # 선택: 참고 문서
└── assets/           # 선택: 템플릿, 리소스

1. SKILL.md (필수)
   - 스킬의 본체
   - frontmatter + 지시사항으로 구성
   - 이것만 있으면 스킬로 인식됨

2. scripts/ (선택)
   - 스킬이 실행할 코드
   - Python, Bash, JavaScript 등
   - 복잡한 처리가 필요할 때 사용

3. references/ (선택)
   - 참고 문서
   - 예: 스타일 가이드, 규칙 문서

4. assets/ (선택)
   - 템플릿, 리소스 파일
   - 예: 출력 템플릿, 이미지

💡 처음에는 SKILL.md만 만들고, 필요할 때 나머지 추가
```

### 3.5 SKILL.md 작성 가이드

```
[SKILL.md 작성 가이드]

1. frontmatter (---로 감싸는 부분)
   ---
   name: skill-name          # 영문 소문자 + 하이픈
   description: 스킬 설명     # 한 문장으로
   triggers:                  # 스킬을 부르는 말
     - "한국어 트리거"
     - "영어 트리거"
     - "/명령어"
   ---

2. 본문 구성 (권장)
   
   # 스킬 이름
   
   ## 목적
   이 스킬이 해결하는 문제
   
   ## 입력
   - 무엇을 받는지
   - 형식, 예시
   
   ## 실행 단계
   1. 첫 번째 단계
   2. 두 번째 단계
   3. ...
   
   ## 출력
   - 무엇을 내놓는지
   - 파일 위치, 형식

3. triggers 작성 팁
   - 다양하게: 한국어, 영어, /명령어 모두 포함
   - 자연스럽게: 실제로 말할 법한 표현으로
   - 예: "뉴스레터 정리", "newsletter", "/newsletter"

4. 로딩 방식
   - 직접 호출: /newsletter-curator
   - 자동 매칭: "뉴스레터 정리해줘" → triggers 매칭
   - 명시적 호출: "newsletter-curator 스킬 실행해줘"
```

### 3.6 스킬 체이닝 개념

```
[스킬 체이닝 개념]

1. 정의
   - 하나의 스킬 결과를 다른 스킬의 입력으로 연결
   - 작은 스킬들을 조합해서 복잡한 워크플로우 구성

2. 비유: 공장 생산라인
   [원자재] → [1공정] → [2공정] → [3공정] → [완성품]
   
   각 공정이 독립적으로 존재하면서,
   앞 공정의 결과물을 다음 공정이 받아서 처리

3. 실전 예시

   예시 A: 뉴스레터 → 주간 리포트
   [Gmail] → [newsletter-curator] → [insight-tagger] → [weekly-digest]
   
   예시 B: 고객 피드백 → 기획
   [문의 수집] → [feedback-classifier] → [priority-scorer] → [prd-drafter]
   
   예시 C: 미팅 → 후속 작업
   [녹취록] → [meeting-summarizer] → [action-extractor] → [task-creator]

4. 설계 원칙
   ① 한 스킬 = 한 가지 일
      - 하나의 스킬이 너무 많은 일을 하지 않도록
      - "이것만 잘하는" 스킬 여러 개가 낫다
   
   ② 입출력 맞추기
      - 스킬 A의 출력 형식 = 스킬 B의 입력 형식
      - md 파일 → md 파일, JSON → JSON
   
   ③ 중간 결과물 저장
      - 각 단계의 결과를 파일로 저장
      - 문제 발생 시 디버깅 용이

5. 실행 방법
   방법 1: 한 번에
   "newsletter-curator 실행하고, 그 결과를 weekly-digest로 넘겨줘"
   
   방법 2: 단계별로
   1. "/newsletter-curator" 실행
   2. 결과 확인
   3. "이 결과를 weekly-digest로 정리해줘"
```

### 3.7 플랜 모드 사용법

```
[플랜 모드 사용법]

1. 플랜 모드란?
   - Claude가 바로 실행하지 않고 계획을 먼저 보여주는 모드
   - 계획을 확인한 후 승인하면 실행

2. 플랜 모드 전환 방법
   - Shift + Tab: 플랜 모드 ON/OFF 토글
   - 또는 자연어로: "플랜부터 만들어줘", "계획만 세워줘"

3. 플랜 모드 활용 흐름
   1) 요청 입력 + Shift+Tab (또는 "플랜부터 만들어줘")
   2) Claude가 계획을 보여줌
   3) 계획 검토 → Clarify로 모호함 제거
   4) 수정된 계획 확인
   5) 승인 → 실행

4. 왜 플랜 모드를 쓰는가?
   - 바로 실행하면 되돌리기 어려움
   - 계획을 먼저 보고 수정할 기회
   - Clarify와 함께 쓰면 효과적

5. 플랜 모드 + Clarify 조합
   플랜 모드로 계획 생성 → /clarify-vague로 모호함 제거 → 수정 → 실행
   이것이 Day 3의 핵심 워크플로우
```

## 7. 비개발자 FAQ

### Q: Clarify가 너무 어려워요
"Clarify는 결국 '질문하기'입니다. Claude에게 '모호한 부분을 질문해줘'라고 하면 Claude가 알아서 질문합니다. 여러분은 그 질문에 답하기만 하면 돼요."

### Q: SKILL.md를 직접 작성해야 하나요?
"아니요! Claude에게 '이런 스킬 만들어줘'라고 하면 Claude가 SKILL.md를 작성해줍니다. 여러분은 결과를 확인하고 수정 요청만 하면 됩니다."

### Q: 스킬이 인식이 안 돼요
"세 가지를 확인해보세요: 1) 폴더 위치가 .claude/skills/ 안에 있는지, 2) SKILL.md 파일이 있는지, 3) frontmatter가 제대로 작성됐는지. 그래도 안 되면 터미널을 재시작해보세요."

### Q: 결과가 매번 달라요
"SKILL.md의 실행 단계가 모호할 가능성이 높습니다. '적절히', '알아서' 같은 표현이 있으면 Claude가 매번 다르게 해석해요. 단계를 더 구체적으로 적어보세요."

### Q: 스킬 체이닝이 잘 이해가 안 돼요
"지금은 개념만 이해하면 됩니다. 실제로 체이닝을 구축하는 건 Day 4~5에서 배웁니다. 오늘은 '스킬끼리 연결할 수 있구나' 정도만 알고 가세요."

### Q: 내 문제가 스킬로 만들기 적합한지 모르겠어요
"Claude에게 물어보세요! '이 문제를 스킬로 만들 수 있을까? [문제 설명]'이라고 하면 Claude가 가능한지, 어떻게 접근하면 좋을지 알려줍니다."

### Q: Git이 뭔지 모르겠어요
"Git은 '게임 세이브'라고 생각하세요. 뭔가 잘못되면 이전 상태로 되돌릴 수 있습니다. Day 4에서 더 자세히 배우니까 오늘은 '실패해도 되돌릴 수 있다' 정도만 알고 계세요."

### Q: Shift+Tab이 안 먹어요
"키보드 설정에 따라 다를 수 있어요. 대안으로 Claude에게 '플랜부터 만들어줘' 또는 '일단 계획만 세워줘, 실행은 하지 마'라고 말하면 같은 효과입니다."
