#### 슬라이드 2-6: Slack — 채널과 스레드
```
Slack
"투명한 채널 기반 소통"

이메일 시대 (Before)        Slack (After)
─────────────────────────────────────────────
받는 사람이 결정 (푸시)  →  받는 사람이 구독 (풀)
CC/BCC 지옥            →  채널 = 맥락의 경계선
답장 체인 꼬임          →  스레드 = 맥락의 깊이

채널 메인 (1층 — 훑어보기)
  Alice: 온보딩 플로우 변경 제안 → 23 replies
  Bob: 점심 12시 반? → 4 replies

스레드 (2층 — 깊이 읽기)
  ├─ Carol: React 기반이 낫지 않나?
  ├─ Dave: 데이터 보니까 이탈율 40%
  └─ Eve: 동의. 다음 스프린트에 넣자

AI에게: 채널 = 무엇이 논의되고 있는지 (스캔)
        스레드 = 왜 그렇게 결정됐는지 (깊이)
```

**상세 내용**:
```
[개념 설명]

Slack의 설계 철학

Slack은 이메일의 "푸시" 방식을 "풀" 방식으로 바꿨습니다.
- 이메일: 보내는 사람이 받는 사람을 결정
- Slack: 받는 사람이 채널을 구독

이 구조가 AI에게 주는 이점:
- 채널 = 맥락의 경계선 (어떤 주제인지 바로 파악)
- 스레드 = 맥락의 깊이 (결정 과정을 따라갈 수 있음)

AI가 "지난주 프로젝트 논의 요약해줘"라고 하면,
채널을 스캔하고 → 스레드를 분석해서 → 핵심만 추출합니다.

Slack의 내부 KPI:
2,000개 메시지가 보내진 워크스페이스는 93%가 리텐션된다.
→ 메시지 사용량 극대화가 핵심
```

#### 슬라이드 2-7: Gmail — 검색 중심
```
Gmail
"정리하지 말고 검색하라" (Search, not Sort)

Outlook (Before)            Gmail (After)
─────────────────────────────────────────────
폴더에 정리해야 한다    →  검색하면 된다
하나의 이메일 = 하나의 폴더 → 하나의 이메일 = 여러 라벨
용량 부족 → 삭제해야 함  →  1GB → 절대 삭제하지 마라
폴더 구조 = 내가 기억해야 → 검색 연산자 = 기계가 찾아줌

AI에게 라벨 = 다차원 메타데이터
  [project-alpha] + [budget] + [from-cfo]
  → AI가 여러 축으로 검색 가능

AI에게 검색 연산자 = 구조화된 쿼리
  from:cfo subject:budget after:2024/01
  → AI가 연산자를 직접 조합해서 검색

AI에게 스레드 = 의사결정 기록
  "Q3 예산이 어떻게 결정됐어?" → 스레드 전체를 분석
```

#### 슬라이드 2-8: Notion — Everything is a Block
```
Notion
"Everything is a Block"

Word/Google Docs            Notion
─────────────────────────────────────────────
텍스트 + 서식 (비구조)   →  타입이 있는 블록 (구조)
"- [ ] 할 일" = 그냥 텍스트 → to_do 블록 = 시맨틱하게 "할 일"
추출하려면 NLP 필요      →  블록 타입으로 바로 쿼리
문서 단위로만 접근       →  블록 단위로 접근 가능

AI에게 블록 = 타입이 있는 구조화된 데이터
  Google Docs: "50페이지에서 할 일 찾아줘" → NLP 필요
  Notion: "to_do 타입 블록 쿼리" → API 한 번

AI에게 데이터베이스 = 필터 가능한 메타데이터
  "Status=진행중 & Owner=Alice"
  → 바로 필터, 파일 하나하나 열 필요 없음

AI에게 Relations = 지식 그래프
  Meeting → Project → Team → OKR
  관계를 따라가면서 맥락을 확장
```

### 3.1 도구별 설계 철학 완전 가이드

```
[도구별 설계 철학 완전 가이드]

1. Slack
   설계 철학: 투명한 채널 기반 소통, 스레드를 통한 맥락 심화
   AI가 읽는 맥락:
   - 채널 = 맥락의 경계선 (무엇이 논의되고 있는지)
   - 스레드 = 맥락의 깊이 (왜 그렇게 결정됐는지)
   
   MCP 연결: Connectors 또는 claude mcp add slack
   주요 도구: slack_read_channel, slack_post_message

2. Gmail
   설계 철학: "정리하지 말고 검색하라" (Search, not Sort)
   AI가 읽는 맥락:
   - 라벨 = 다차원 메타데이터 (여러 기준으로 분류)
   - 검색 연산자 = 구조화된 쿼리 (from:, subject:, after:)
   - 스레드 = 의사결정 기록
   
   연결: API 스크립트 (Claude가 작성)

3. Notion
   설계 철학: "Everything is a Block"
   AI가 읽는 맥락:
   - 블록 = 타입이 있는 구조화된 데이터
   - 데이터베이스 = 필터 가능한 메타데이터
   - Relations = 지식 그래프
   
   MCP 연결: 
   - Connectors (가장 쉬움)
   - claude mcp add --transport http notion https://mcp.notion.com/mcp
   
   MCP 월간 검색량: 9,500회 (비개발 도구 중 1위)

4. Google Drive
   설계 철학: 파일 공유 + 실시간 공동 편집 + 클라우드
   AI가 읽는 맥락:
   - 축적된 산출물 아카이브 (비구조화된 파일)
   - 버전 히스토리
   
   연결: Connectors 또는 API 스크립트

5. Linear/Jira
   설계 철학: 애자일 방법론 구현, 작업 추적 및 실행 구조화
   AI가 읽는 맥락:
   - 이슈 = 구조화된 데이터 (실행 상태)
   - 스프린트 = 시간 범위가 있는 작업 묶음
   
   MCP 연결: Connectors 또는 명령어

6. Google Calendar
   설계 철학: 미래의 시간 자원 구조화
   AI가 읽는 맥락:
   - 이벤트 = 미래의 구조화된 데이터
   - 크로스 도구 검색의 시작점
   
   연결: API 스크립트 (Claude가 작성)
   
   특별함: 다른 모든 도구는 과거/현재를 담지만,
          Calendar만 미래를 담는다.
```

### 3.2 MCP 완전 가이드

```
[MCP (Model Context Protocol) 완전 가이드]

1. 정의
   - AI가 외부 서비스와 대화하는 표준 규약
   - USB-C처럼 다양한 도구를 표준 방식으로 연결
   - Claude가 Slack, Notion, Gmail을 같은 방식으로 대화

2. 작동 원리
   사용자: "Slack 메시지 읽어줘"
       │
       ▼
   Claude: "Slack MCP 서버에 요청"
       │
       ▼ MCP 프로토콜 (표준 규격)
       │
   Slack MCP 서버: 메시지 조회 API 호출
       │
       ▼
   Claude: 결과를 사용자에게 전달

3. 연결 방법 3가지
   
   방법 1: Connectors (가장 쉬움) ⭐
   - claude.ai/settings/connectors 에서 클릭으로 연결
   - 코드 한 줄 없이 버튼만 누르면 됨
   - 지원: Slack, Notion, Google Drive, Linear 등
   
   방법 2: 명령어
   - claude mcp add --transport http notion https://mcp.notion.com/mcp
   - claude mcp add slack
   - 터미널에서 한 줄이면 연결 완료
   
   방법 3: 수동 설정 (.mcp.json)
   - 프로젝트 루트에 .mcp.json 파일 직접 편집
   - 세밀한 설정이 필요할 때 사용

4. 확인 방법
   /mcp
   - 연결된 서버 목록
   - 사용 가능한 도구(Tools) 확인

5. 관리 명령어
   claude mcp disable [서버명]  # 비활성화
   claude mcp enable [서버명]   # 활성화
   
6. 컨텍스트 관리 팁
   - MCP 도구가 많으면 컨텍스트 토큰 많이 소비
   - 자주 쓰는 도구만 연결
   - 안 쓰는 MCP는 disable
   - Lazy Loading: 필요할 때만 연결 (고급 설정)

7. 모범 사례
   ✅ 자주 쓰는 도구만 연결
   ✅ /mcp로 연결 상태 주기적 확인
   ✅ 문제 있으면 disable → 다시 enable
   
   ❌ 모든 도구를 한꺼번에 연결
   ❌ 사용하지 않는 연결 방치
```

### 3.3 Context Sync 스킬 완전 가이드

```
[Context Sync 스킬 완전 가이드]

1. 정의
   - 여러 도구에서 정보를 자동 수집하여 하나의 문서로 정리
   - "여러 앱 돌아다니는 30분을 '싱크해줘' 1분으로"

2. 스킬 구조
   .claude/skills/my-context-sync/
   ├── SKILL.md          # 스킬 본체
   ├── scripts/          # 수집 스크립트 (Gmail, Calendar 등)
   │   ├── gmail_fetch.py
   │   └── calendar_fetch.py
   └── references/       # 참고 문서

3. SKILL.md 구조
   ---
   name: my-context-sync
   description: 나의 컨텍스트 싱크. "싱크", "sync" 요청에 사용.
   triggers:
     - "싱크"
     - "sync"
     - "컨텍스트 싱크"
   ---
   
   # My Context Sync
   
   ## 소스 정의
   - Slack: #general, #project-updates
   - Notion: 업무 태스크 DB
   - Calendar: 오늘 ~ 7일 후
   
   ## 실행 흐름
   1. 병렬 수집 (Subagent)
   2. 결과 통합
   3. 문서 저장
   
   ## 출력
   sync/YYYY-MM-DD-context-sync.md

4. 직종별 추천 도구 조합
   - PM/기획자: Slack + Linear/Jira + Calendar
   - 마케터: Slack + Gmail + 소셜미디어
   - 영업: Gmail + Calendar + CRM
   - CEO/리더: Slack + Calendar + 뉴스/RSS
   - 디자이너: Slack + Notion + Figma
   - 개발자: Slack + GitHub + Linear

5. 구축 단계 (Block 0~5)
   Block 0: 도구 선택 + 스킬 골격 생성
   Block 1: Explore 에이전트로 프로젝트 탐색
   Block 2: MCP/API로 도구 연결
   Block 3: Subagent로 병렬 수집 + 검증
   Block 4: Output 형식 설정 (Markdown/Slack/Notion)
   Block 5: 완성 + 실행 테스트

6. 활용 팁
   - 매일 아침 실행 → 모닝 브리핑
   - CLAUDE.md에 스케줄 추가 → 자동 실행
   - 소스 추가는 스킬 파일 수정으로 언제든 가능
   - Day 3 이후 더 복잡한 워크플로우로 발전 가능
```

### 3.4 Subagent 병렬 수집

```
[Subagent 병렬 수집]

1. Subagent란?
   - 메인 Claude가 다른 Claude를 불러서 작업 위임
   - 비유: 부하 직원에게 일 시키기
   - 독립된 컨텍스트에서 작업 후 결과만 반환

2. 병렬 수집이 가능한 이유
   - Slack, Notion, Calendar는 서로 의존성 없음
   - 각각 독립적으로 수집 가능
   - 동시에 실행하면 시간 절약

3. 작동 방식
   사용자: "싱크해줘"
       │
       ▼
   메인 Claude: "3개 Subagent 실행"
       │
       ├─ Subagent 1 ─── Slack 수집
       ├─ Subagent 2 ─── Notion 수집
       └─ Subagent 3 ─── Calendar 수집
       │
       ▼ (3개 동시 실행)
       │
   메인 Claude: 결과 통합 → 문서 생성

4. 주의 사항
   - 각 Subagent 결과가 너무 크면 정보 손실 가능
   - 추천: Subagent가 파일로 저장 → 메인이 파일 읽기
   - 토큰 효율을 위해 요약만 반환하도록 설정

5. Context Sync에서의 활용
   "Task 도구로 subagent 실행:
    - Slack: 최근 7일 메시지 수집
    - Notion: 진행 중 태스크 수집
    - Calendar: 이번 주 일정 수집
    
    각 결과를 파일로 저장 후 통합"
```
